# AWS Athena란 무엇인가?

AWS Athena는 S3에 저장된 파일을 테이블처럼 보고 SQL로 조회할 수 있게 해주는 서버리스 분석 서비스입니다.

데이터를 RDBMS에 따로 넣지 않고, S3에 있는,

- csv
- json
- parquet
- orc
- avro

같은 파일을 대상으로 SQL을 실행할 수 있는 서비스입니다.

AWS 공식 설명에서도 Athena는 S3 데이터를 표준 SQL로 분석할 수 있는 interactive query service
이며, 서버를 직접 구성하거나 관리할 필요가 없다고 설명합니다.

## Athena가 필요한 이유

일반적으로 데이터를 분석하려면 다음과 같은 작업이 필요합니다.

```shell
데이터 수집
→ DB 또는 Data Warehouse에 적재
→ 테이블 생성
→ 쿼리 실행
→ 결과 분석
```

하지만 Athena를 사용하면 중간 과정이 단순해집니다.

```shell
S3에 데이터 저장
→ Athena에서 테이블 정의
→ SQL 실행
→ 결과 확인
```

## Athena의 핵심 특징

- 서버리스
  - EC2, RDS, Redshift 클러스터를 직접 운영하지 않아도 됨

- SQL 사용
  - 표준 SQL 기반으로 S3 데이터를 조회 가능

- S3 직접 조회
  - 데이터를 별도 DB에 적재하지 않고 S3 파일을 바로 분석

- 사용량 기반 과금
  - 실행한 쿼리에서 스캔한 데이터 양 또는 사용한 컴퓨팅 기준으로 과금

- 자동 확장
  - 쿼리 실행 시 필요한 리소스를 Athena가 자동으로 처리

- Glue 연동
  - AWS Glue Data Catalog를 메타데이터 저장소로 사용할 수 있음

> 👉 AWS 공식 페이지에서도 Athena는 인프라 설정 없이 사용할 수 있고, 쿼리를 병렬로 실행하여
> 큰 데이터셋도 빠르게 처리하도록 설계되어 있다고 설명합니다.

## Athena는 어떻게 동작하는가?

Athena의 동작 흐름은 다음과 같습니다.

```shell
1. S3에 데이터 파일 저장
2. Glue Data Catalog 또는 Athena DDL로 테이블 스키마 정의
3. Athena에서 SQL 작성
4. Athena가 S3 파일을 읽어서 쿼리 실행
5. 쿼리 결과를 S3에 저장
6. 콘솔, API, JDBC, ODBC 등으로 결과 조회
```

구조로 보면 다음과 같습니다.

```shell
사용자 / 애플리케이션
        ↓
    Amazon Athena
        ↓
AWS Glue Data Catalog
        ↓
    Amazon S3
```

> 👉 여기서 중요한 점은 Athena 자체가 데이터를 저장하는 DB는 아니라는 것입니다.
> 실제 데이터는 S3에 있고, Athena는 그 데이터를 읽어서 SQL을 실행하는 쿼리 엔진에 가깝습니다.

## Parquet

Parquet은 컬럼 기반 저장 형식이며, Athena에서 성능과 비용 측면에서 유리합니다.

CSV나 JSON은 전체 행을 많이 읽어야 하는 경우가 많지만, Parquet은 컬럼 기반 포맷이라 필요한 컬럼만
효율적으로 읽을 수 있습니다. Athena는 스캔한 데이터 양을 기준으로 과금되는 구조이기 때문에,
Parquet을 사용하면 성능과 비용 모두에서 유리합니다.

## Athena와 RDS의 차이

| 구분             | Athena                               | RDS                               |
| ---------------- | ------------------------------------ | --------------------------------- |
| 목적             | 분석 쿼리                            | 애플리케이션 운영 DB              |
| 데이터 저장 위치 | S3                                   | DB 인스턴스 내부 스토리지         |
| 서버 관리        | 필요 없음                            | 인스턴스 관리 필요                |
| 쿼리 방식        | S3 파일 대상 SQL                     | 테이블 데이터 대상 SQL            |
| 적합한 작업      | 로그 분석, 통계, 리포트, ad-hoc 분석 | 회원, 주문, 결제 등 트랜잭션 처리 |
| 실시간 CRUD      | 부적합                               | 적합                              |
| 과금 방식        | 주로 쿼리 스캔량 기반                | 인스턴스, 스토리지, I/O 기반      |

## Athena의 대표 사용 사례

### 1. 로그 분석

예를 들어 다음과 같은 로그를 S3에 저장해두고 Athena로 조회할 수 있습니다.

```shell
- ALB Access Log
- CloudFront Log
- VPC Flow Log
- CloudTrail Log
- Application Log
- Search Log
- Click Event Log
```

이런 식으로 특정 날짜의 HTTP 상태 코드별 요청 수를 분석:

```sql
SELECT
    status_code,
    COUNT(*) AS request_count
FROM alb_access_logs
WHERE year = 2026
  AND month = 5
  AND day = 27
GROUP BY status_code
ORDER BY request_count DESC;
```

### 2. 검색 로그 분석

쇼핑몰이나 서비스에서 검색어 로그를 S3에 쌓아두고 Athena로 분석할 수 있습니다.

```json
{"query": "sunscreen", "took_ms": 120, "is_zero_result": false}
{"query": "vitamin c serum", "took_ms": 85, "is_zero_result": false}
{"query": "unknown product", "took_ms": 45, "is_zero_result": true}
```

### 3. 데이터 레이크 분석

S3를 데이터 레이크로 사용하고 있다면 Athena는 매우 자연스럽게 붙일 수 있습니다.

```shell
- 서비스 로그
- 주문 데이터
- 클릭 이벤트
- 검색 이벤트
- 상품 데이터
- 광고 성과 데이터
```

### 4. 임시 분석 작업

운영 DB에 부하를 주기 싫을 때, 데이터를 S3로 내보낸 뒤 Athena로 분석할 수 있습니다.
예를 들어 다음과 같은 경우입니다.

```shell
- 최근 3개월 주문 통계 분석
- 이벤트 기간 사용자 행동 분석
- 특정 검색어의 일별 검색량 분석
- 상품별 클릭 수 분석
```

운영 DB에 무거운 GROUP BY, COUNT, JOIN 쿼리를 직접 날리는 것보다, S3에 데이터를 적재하고
Athena로 분석하는 구조가 더 안전할 수 있습니다.

## Athena의 장점과 단점

### 장점

| 장점                    | 설명                                             |
| ----------------------- | ------------------------------------------------ |
| 인프라 관리가 필요 없음 | 서버, 클러스터, 패치, 확장 관리를 직접 하지 않음 |
| 시작이 빠름             | S3 데이터와 스키마만 있으면 바로 쿼리 가능       |
| SQL 기반                | 개발자와 데이터 분석가가 익숙한 SQL 사용 가능    |
| S3 데이터 직접 분석     | 데이터를 별도 저장소로 복사하지 않아도 됨        |
| 비용 효율적             | 사용한 만큼만 비용 발생                          |
| AWS 서비스와 연동 좋음  | S3, Glue, QuickSight, Lambda, IAM 등과 연동 가능 |

### 단점

| 단점                              | 설명                                                       |
| --------------------------------- | ---------------------------------------------------------- |
| 실시간 트랜잭션 처리에는 부적합   | INSERT, UPDATE, DELETE 중심의 서비스 DB 용도로는 맞지 않음 |
| 쿼리 성능이 파일 구조에 크게 의존 | 파티션, 파일 포맷, 압축 여부에 따라 성능 차이가 큼         |
| 잘못 사용하면 비용 증가           | 큰 CSV/JSON 파일을 반복 조회하면 스캔 비용이 커질 수 있음  |
| 작은 파일이 너무 많으면 비효율적  | S3에 작은 파일이 과도하게 많으면 성능 저하 가능            |
| 복잡한 OLTP 업무에는 부적합       | 주문 처리, 결제 처리 같은 운영성 업무에는 RDS가 적합       |

## Athena 과금 방식

- Athena는 기본적으로 쿼리가 스캔한 데이터 양을 기준으로 과금됩니다.
- AWS 공식 가격 페이지에서는 Athena가 S3의 데이터를 직접 조회하며, 쿼리 결과도 사용자가 지정한
  S3 버킷에 저장되고 해당 S3 비용은 별도로 부과된다고 설명합니다.
- 또한 Glue Data Catalog를 사용하는 경우 Glue 요금이 발생할 수 있습니다.

즉, Athena 비용을 줄이려면 핵심은 하나입니다. 쿼리가 읽는 데이터 양을 줄여야 한다.

### Athena 비용 최적화 방법

#### 1. Parquet 사용하기

CSV나 JSON보다 Parquet을 사용하는 것이 유리합니다.

```shell
CSV / JSON
→ 행 기반
→ 필요 없는 컬럼까지 많이 읽을 수 있음

Parquet
→ 컬럼 기반
→ 필요한 컬럼만 효율적으로 읽을 수 있음
```

#### 2. 파티션 사용하기

날짜별 데이터라면 다음과 같이 S3 경로를 나누는 것이 좋습니다.

```shell
s3://my-bucket/search-log/year=2026/month=05/day=27/
s3://my-bucket/search-log/year=2026/month=05/day=26/
s3://my-bucket/search-log/year=2026/month=05/day=25/
```

이렇게 하면 Athena가 전체 데이터를 읽지 않고 필요한 파티션만 읽을 수 있습니다.

#### 3. 필요한 컬럼만 조회하기

비용을 줄이려면 SELECT \* FROM... 을 피하는 것이 좋습니다.

#### 4. 압축 사용하기

S3에 저장되는 데이터를 압축하면 Athena가 읽는 데이터 양을 줄일 수 있습니다.

```shell
gzip
snappy
zstd
```

## Athena가 적합한 경우

| 상황                                     | 적합 여부 |
| ---------------------------------------- | --------- |
| S3에 로그가 쌓여 있다                    | 매우 적합 |
| 검색어 통계를 분석하고 싶다              | 적합      |
| 클릭 이벤트를 일별로 집계하고 싶다       | 적합      |
| 운영 DB에 부하 없이 리포트를 만들고 싶다 | 적합      |
| 가끔 대용량 데이터를 분석한다            | 적합      |
| 매초 수천 건의 INSERT/UPDATE가 필요하다  | 부적합    |
| 결제/주문 같은 트랜잭션 처리가 필요하다  | 부적합    |
| 밀리초 단위 응답이 필요하다              | 부적합    |

## 정리

AWS Athena는 S3에 저장된 데이터를 SQL로 직접 분석할 수 있는 서버리스 쿼리 서비스입니다.

| 핵심 포인트   | 설명                                                |
| ------------- | --------------------------------------------------- |
| 서비스 성격   | S3 데이터 분석용 서버리스 SQL 쿼리 서비스           |
| 데이터 저장소 | Athena가 아니라 S3                                  |
| 메타데이터    | 주로 AWS Glue Data Catalog 사용                     |
| 주요 장점     | 인프라 관리 없음, 빠른 시작, SQL 사용, S3 직접 조회 |
| 주요 단점     | 실시간 트랜잭션 처리에는 부적합                     |
| 비용 기준     | 주로 쿼리가 스캔한 데이터 양                        |
| 최적화 핵심   | Parquet, 파티션, 압축, 필요한 컬럼만 조회           |

> 👉 Athena는 S3를 단순한 파일 저장소가 아니라 SQL로 분석 가능한 데이터 레이크로 활용하게 해주는
> AWS의 서버리스 분석 서비스다.
