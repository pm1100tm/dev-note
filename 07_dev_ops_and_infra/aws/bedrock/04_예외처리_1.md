# 예외 처리

설계: callBedrock 예외 처리 best practice

현재 상황 분석

AiTranslationAdapter#callBedrock은 AWS Bedrock SDK 예외(ThrottlingException, ModelTimeoutException, ServiceQuotaExceededException, BedrockRuntimeException, SdkException 등)를 그대로 흘려보내고 있습니다.
그러면 GlobalExceptionHandler의 catch-all Exception 핸들러가 잡아서:

- HTTP 500을 응답
- ex.getMessage()(SDK가 노출하는 내부 메시지)를 그대로 프론트에 전달

→ 사용자에게 인프라 디테일이 새고, 어떤 에러인지 의미적으로 구분도 안 됩니다.

기존 컨벤션에서 가져올 4가지 패턴

코드베이스를 확인한 결과, 이 프로젝트는 다음 컨벤션을 일관되게 따르고 있습니다:

1. Adapter는 SDK 예외를 application 예외로 변환 — adapter 경계에서 외부 라이브러리 예외가 더 위로 새지 않도록 막음 (ApplicationException이 inbound adapter에 노출되는 boundary type이라고 javadoc에 명시되어
   있음)
2. TranslatableMessage 기반 i18n — ApplicationException/DomainException은 모두 TranslatableMessage를 받고, messages_en.properties에 키를 등록 (예: member.login.required, product.access.notfound)
3. 정적 팩토리 메서드 패턴 — UnauthorizedException.invalidState(), ProductAccessDeniedException.notFound(...)처럼 의미적 카테고리별로 정적 팩토리 제공 (생성자 private)
4. GlobalExceptionHandler에서 HTTP status 매핑 — 도메인/애플리케이션 예외 타입 → HTTP status. resolveCode가 TranslatableMessage의 messageKey()를 응답 code로 사용하므로 프론트가 코드로 분기 가능

스택트레이스는 이미 ErrorResponse(status, code, message, timestamp) record에 포함되지 않으므로(👍) 노출되지 않습니다. 핵심은 ex.getMessage()가 SDK 내부 메시지가 아니도록 막는 것입니다.

설계 결정

(A) 새 예외 타입: AiTranslationException extends ApplicationException

위치: application/exception/AiTranslationException.java

public class AiTranslationException extends ApplicationException {

      private AiTranslationException(TranslatableMessage message) {
          super(message);
      }

      /** 일시적 과부하 / rate limit — 사용자가 잠시 후 재시도 가능 */
      public static AiTranslationException throttled() {
          return new AiTranslationException(TranslatableMessage.from(
              "ai.translation.throttled",
              "Translation service is busy. Please try again in a moment."));
      }

      /** 모델 타임아웃 — 입력이 너무 길거나 일시 지연 */
      public static AiTranslationException timedOut() {
          return new AiTranslationException(TranslatableMessage.from(
              "ai.translation.timeout",
              "Translation took too long. Please try again."));
      }

      /** 그 외 모든 실패 (네트워크/내부 오류/검증 실패) — 일반 fallback */
      public static AiTranslationException unavailable() {
          return new AiTranslationException(TranslatableMessage.from(
              "ai.translation.unavailable",
              "Translation service is temporarily unavailable. Please try again later."));
      }

}

카테고리를 3가지로 나눈 이유: 사용자에게 정확한 원인은 숨기되, 프론트가 행동을 분기할 수 있는 최소한의 정보(retry-soon vs retry-later vs unavailable)는 줍니다. 기존 UnauthorizedException이 invalidState() /
providerMismatch()로 나뉜 것과 동일한 입도입니다.

(B) Adapter에서 SDK 예외 → 애플리케이션 예외 변환

AiTranslationAdapter#callBedrock을 다음처럼 감쌉니다. try-catch는 boundary(adapter)에서 한 번만, 호출 위치마다 흩어 놓지 않습니다.

private String callBedrock(String prompt, PromptConfig config) {
Message userMessage = Message.builder()
.role(ConversationRole.USER)
.content(ContentBlock.fromText(prompt))
.build();

      try {
          ConverseResponse response = bedrockClient.converse(request -> request
              .modelId(properties.getModelId())
              .messages(userMessage)
              .inferenceConfig(cfg -> cfg
                  .temperature(config.temperature())
                  .maxTokens(config.maxTokens()))
          );
          return response.output().message().content().getFirst().text();

      } catch (ThrottlingException e) {
          log.warn("[AI-TRANSLATION] Bedrock throttled: model={}, msg={}",
              properties.getModelId(), e.getMessage(), e);
          throw AiTranslationException.throttled();

      } catch (ModelTimeoutException e) {
          log.warn("[AI-TRANSLATION] Bedrock model timeout: model={}, promptLen={}, msg={}",
              properties.getModelId(), prompt.length(), e.getMessage(), e);
          throw AiTranslationException.timedOut();

      } catch (ServiceQuotaExceededException | ModelNotReadyException
               | ServiceUnavailableException e) {
          log.error("[AI-TRANSLATION] Bedrock unavailable: model={}, type={}, msg={}",
              properties.getModelId(), e.getClass().getSimpleName(), e.getMessage(), e);
          throw AiTranslationException.unavailable();

      } catch (ValidationException | ModelErrorException | AccessDeniedException
               | ResourceNotFoundException e) {
          // 운영자 액션이 필요한 종류 (입력 검증 / 권한 / 모델 설정 오류).
          // 사용자에겐 동일하게 unavailable로 보여주고, 로그로 추적.
          log.error("[AI-TRANSLATION] Bedrock config/input error: model={}, type={}, msg={}",
              properties.getModelId(), e.getClass().getSimpleName(), e.getMessage(), e);
          throw AiTranslationException.unavailable();

      } catch (SdkException e) {
          // 네트워크 I/O, 타임아웃 등 모든 기타 AWS SDK 예외 — 최후의 fallback.
          log.error("[AI-TRANSLATION] Bedrock call failed: model={}, msg={}",
              properties.getModelId(), e.getMessage(), e);
          throw AiTranslationException.unavailable();
      }

}

원칙:

- 로그에는 full stack trace + 운영 컨텍스트(model id, prompt length, exception type) 기록 — 디버깅 / 모니터링 / 알람용
- 응답에는 TranslatableMessage로 정제된 메시지만 — SDK의 e.getMessage()는 응답에 절대 포함되지 않음
- SdkException을 마지막 catch로 — BedrockRuntimeException까지 모두 포함하므로 누락 없음

(C) GlobalExceptionHandler에 핸들러 추가

@ExceptionHandler(AiTranslationException.class)
public ResponseEntity<ErrorResponse> handleAiTranslation(AiTranslationException ex) {
return buildResponse(ex, HttpStatus.SERVICE_UNAVAILABLE); // 503
}

기존 buildResponse 흐름을 그대로 타기 때문에:

- resolveCode(ex)는 TranslatableException 분기에 걸려 messageKey() 즉 ai.translation.throttled / ai.translation.timeout / ai.translation.unavailable을 응답 code로 내보냄
- resolveMessage(ex)는 MessageTranslator로 i18n 메시지 반환
- logException이 5xx이므로 log.error로 한 번 더 기록 (adapter에서 이미 기록했으므로 중복이지만 표준 패턴 유지)

(D) i18n 메시지 등록

src/main/resources/i18n/messages_en.properties (그리고 \_ru.properties):

ai.translation.throttled=Translation service is busy. Please try again in a moment.
ai.translation.timeout=Translation took too long. Please try again.
ai.translation.unavailable=Translation service is temporarily unavailable. Please try again later.

응답 예시

기존 ErrorResponse 구조 그대로 사용 (스택트레이스 없음):

{
"status": 503,
"code": "ai.translation.throttled",
"message": "Translation service is busy. Please try again in a moment.",
"timestamp": "2026-06-03T23:51:34"
}

프론트는 code 필드로 카테고리를 식별하고, message를 그대로 사용자에게 노출.

변경 영향 범위

- 신규 파일: AiTranslationException.java (1개)
- 수정: AiTranslationAdapter.java (callBedrock만), GlobalExceptionHandler.java (핸들러 1개 추가), messages_en.properties, messages_ru.properties (각 3줄 추가)
- 도메인 레이어 / 다른 adapter / 다른 use case에 영향 없음
