# EXPLAIN vs EXPLAIN ANALYZE 차이점

## 개념 비교

| 구분      | EXPLAIN                 | EXPLAIN ANALYZE                  |
| --------- | ----------------------- | -------------------------------- |
| 실행 여부 | 쿼리를 실행하지 않음    | 쿼리를 실제로 실행함             |
| 결과 종류 | 옵티마이저의 예측값     | 예측값 + 실제 측정값             |
| 소요 시간 | 즉시 반환               | 쿼리 실행 시간만큼 소요          |
| 부작용    | 없음                    | SELECT는 안전, DML은 실제 반영됨 |
| 용도      | 대략적인 실행 계획 파악 | 예측과 실제의 괴리를 파악        |

<br>

## 핵심 차이: 예측 vs 실측

```
-- EXPLAIN: 옵티마이저가 "이렇게 실행할 것"이라는 계획서
EXPLAIN SELECT ...;

-- EXPLAIN ANALYZE: 실제로 실행해보고 "이렇게 실행됐음"을 보고
EXPLAIN ANALYZE SELECT ...;

EXPLAIN ANALYZE 결과에 추가되는 핵심 정보:

(cost=예측비용 rows=예측행수) (actual time=시작ms..종료ms rows=실제행수 loops=반복횟수)

- cost: 옵티마이저가 예측한 상대적 비용
- rows: 옵티마이저가 예측한 처리 행 수
- actual time: 해당 노드의 첫 번째 행..마지막 행까지 실제 소요 시간(ms)
- loops: 해당 노드가 반복 실행된 횟수
- 실제 총 시간 = actual time의 두 번째 값 × loops

언제 무엇을 쓰나?

- EXPLAIN: 인덱스 사용 여부, 조인 순서를 빠르게 확인할 때
- EXPLAIN ANALYZE: 느린 쿼리의 실제 병목 지점을 찾을 때 (운영 DB에서 무거운 쿼리 실행 시 주의)
```

<br>

## 실행 계획 읽는 방법 (트리 구조 기준)

MySQL 8.0의 EXPLAIN ANALYZE는 트리 형태로 출력된다. 읽는 순서가 중요하다.

### 읽기 규칙

- 가장 안쪽(들여쓰기 깊은 곳) = 가장 먼저 실행
- 바깥쪽으로 나올수록 = 나중에 실행

실행 계획 트리를 아래에서 위로, 안쪽에서 바깥쪽으로 읽는다.

### 핵심 판단 지표

#### rows 예측 vs 실제 괴리

```sql
(cost=0.45 rows=1) (actual time=... rows=2299 loops=1)
^^^^^^ ^^^^^
예측: 1행 실제: 2299행 ← 통계 부정확
```

예측과 실제의 차이가 클수록 옵티마이저가 잘못된 판단을 내릴 가능성이 높다.

#### loops × actual time = 실제 누적 비용

```sql
(actual time=290e-6..363e-6 rows=0.999 loops=2299)
실제 총 시간 = 363e-6 ms × 2299 = 약 0.83ms
```

loops가 크면 실제 비용이 숫자보다 훨씬 크다.

#### 접근 방식 (scan type)

| 접근 방식           | 의미                                                   | 위험도  |
| ------------------- | ------------------------------------------------------ | ------- |
| Covering Index Scan | 인덱스만으로 쿼리를 처리하여 테이블 접근이 필요 없음   | 낮음 ✅ |
| Index Lookup        | 인덱스로 행의 위치를 찾은 뒤 실제 테이블 데이터에 접근 | 낮음 ✅ |
| Table Scan          | 테이블의 모든 행을 처음부터 끝까지 읽음                | 높음 ⚠️ |
| Full Index Scan     | 인덱스 전체를 처음부터 끝까지 읽음                     | 중간    |
